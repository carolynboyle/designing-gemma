# Design Doc: manifest_filter.py
## designing-gemma

---

## Problem Statement

`doc-gen` generates a `manifest.yml` covering an entire repository. For
repos like `dev-utils` that contain bash scripts, container configs, and
Python packages under separate top-level directories, the full manifest
is too broad for experiments that only care about a subset.

Feeding a 200-file manifest through `skeleton_reader` and into a 40,000
character prompt budget means Gemma receives bash scripts and Dockerfiles
before it ever sees the Python packages it was asked to analyze. The
character budget is exhausted on irrelevant content.

The fix is a separate filtering step that produces a scoped manifest from
an existing one. `skeleton_reader` then reads the scoped manifest instead
of the full one.

---

## Design

`manifest_filter.py` is a simple, single-purpose module. It reads a
`manifest.yml`, filters entries by path prefix, and writes a new
manifest to a specified output path.

It does not walk the filesystem. It does not modify the source manifest.
It does not know about skeletons, prompts, or experiments. It filters
a list and writes a file.

---

## CLI

```bash
designing-gemma-filter <repo_path> --include <prefix> [--include <prefix> ...] [--output <path>]
```

**Arguments:**

- `repo_path` — path to the target repo (must contain `.doc-gen/manifest.yml`)
- `--include` — path prefix to include (repeatable, OR logic — any match is kept)
- `--output` — output path relative to repo root (default: `.doc-gen/manifest.filtered.yml`)

**Examples:**

```bash
# Keep only Python packages
designing-gemma-filter /repos/dev-utils --include python/

# Keep Python packages and bash scripts
designing-gemma-filter /repos/dev-utils --include python/ --include bash/

# Write to a named output
designing-gemma-filter /repos/dev-utils --include python/ --output .doc-gen/manifest.python.yml
```

---

## Pipeline Position

```
doc-gen manifest.yml  (full repo, generated by doc-gen)
       ↓
designing-gemma-filter  (keep only python/*)
       ↓
manifest.python.yml  (scoped manifest)
       ↓
skeleton_reader  (reads scoped manifest, writes manifest.skel)
       ↓
repo_reader  (reads manifest.skel, formats for prompt injection)
       ↓
{{ repo_context }} in prompt
       ↓
Gemma
```

`doc-gen` and `manifest_filter` are both pre-run steps. The runner
calls `skeleton_reader` and `repo_reader` at runtime using whatever
manifest the experiment config points at.

---

## Output Format

The filtered manifest follows the same format as `doc-gen`'s
`manifest.yml` so it is a drop-in replacement:

```yaml
# Generated by designing-gemma manifest_filter
# Source: .doc-gen/manifest.yml
# Filter: python/
# Generated: 2026-05-13 14:00:00
# Files: 47

version: 1
documents:
  - path: python/fletcher/README.md
  - path: python/fletcher/__init__.py
  - path: python/fletcher/fletcher.py
  ...
```

## Integration with repo_read config

After filtering, point the experiment's `repo_read` block at the
filtered manifest:

```yaml
repo_read:
  repo: dev_utils
  manifest: .doc-gen/manifest.python.yml
  max_chars: 40000
  size_overrides:
    - python/fletcher/README.md
```

`skeleton_reader` reads whatever manifest path it is given. No changes
to `skeleton_reader` or `repo_reader` are needed.

## Module Design

**Location:** `src/designing_gemma/manifest_filter.py`

**Public function:**

```python
def filter_manifest(
    repo_path: Path,
    include_prefixes: list[str],
    source_manifest: str = ".doc-gen/manifest.yml",
    output_manifest: str = ".doc-gen/manifest.filtered.yml",
) -> dict:
    """
    Read source manifest, filter by path prefixes, write output manifest.

    Args:
        repo_path:        Root path of the target repo.
        include_prefixes: List of path prefixes to include (OR logic).
                          e.g. ["python/", "bash/"]
        source_manifest:  Path to source manifest.yml relative to repo_path.
        output_manifest:  Path to write filtered manifest relative to repo_path.

    Returns:
        Dict with keys:
            total        — total entries in source manifest
            included     — entries kept after filtering
            excluded     — entries removed by filter
            output_path  — absolute path to written file

    Raises:
        ManifestFilterError: If source manifest cannot be read or parsed.
    """
```

**Filter logic:** simple `str.startswith()` check against each prefix.
An entry is included if its path starts with ANY of the specified prefixes.
Case-sensitive — paths in manifests are always lowercase on Linux.

**Error class:** `ManifestFilterError(Exception)` — raised on missing or
malformed source manifest.

## Notes for fileskit

When this logic moves to dev-utils as a standalone `fileskit` tool:

- Remove the designing-gemma package dependency entirely — pure stdlib
  (pathlib + yaml)
- Accept multiple manifest formats if other tools produce them
- Consider adding `--exclude` flag for negative filtering
- Consider glob pattern support beyond simple prefix matching
  (e.g. `**/*.py` to filter by file type rather than directory)
- The `filter_manifest()` function is already close to standalone —
  it has no designing-gemma-specific dependencies beyond `yaml`
- A `--dry-run` flag showing what would be included/excluded without
  writing would be useful for verifying filter patterns before committing
  to them
- Could also serve as a manifest merge tool — combine two manifests
  into one, deduplicated — useful when you want Python files from one
  repo and config files from another
